What We Know

1 Mixins

Mixins are ‘functions’ that return css.

      
  //scss
  @mixin my-mixn($color, $size) {
    color: $color;
    font-size: $size;
  }   
  .myclass{
    @include my-mixn(red, 20px);
  }

  // css 
  .myclass {
    color: red;
    font-size: 20px;
  }

2 Functions

Functions are like mixins but they return a value, not css


  //scss
  @function mycalc($arg:10px){
    @return $arg + 20px;
  }

  .myclass {
    height: mycalc(30px);
  }

  //css
  .myclass {
    height: 50px;
  }

3 Variables

Variables begin with a dollar sign.


  //scss
  $primary-color: #0078A0; 

  .myclass {
  color: rgba($primary-color, .5);
  }

  // css
 .myclass {
    color: rgba(0, 120, 160, 0.5);
  }

5 Partials

Partials are named with an underscore. ( _uname.scss , for example), but the underscore is ignored when importing the file.

6 Extend

The h2 selector extends h1 


  //scss
  h1 {
      color:red;
      background-color: green;
    }

  h2 {
      @extend h1;
      font-size: 20px;
    }

  //css
  h1, h2 {
    color: red;
    background-color: green;
  }

  h2 {
    font-size: 20px;
  }

Extend with placeholder (base class)


  //scss
  %heading-base {
    color:red;
    background-color: green;
  }

  h1 {
    @extend %heading-base;
  }

  h2 {
    @extend %heading-base;
    font-size: 20px;
  }

  //css
  h1, h2 {
    color: red;
    background-color: green;
  }
  h2 {
    font-size: 20px;
  }

7 Ampersand

The & references the parent selector. There are two basic types of use:


  //scss
  div {
    &.myclass {
    background-color: red;   
      }
   }

  //css 
  div.myclass {
    background-color: red;
  }

  and

  //scss
  div {
    .myclass2 & {
      background-color: red;   
    }
  }

  //css
   .myclass2 div {
    background-color: red;
  }

The following also works (since Sass 3.2?) where &-new expands to .myclass-new 


  //scss
  .myclass3 {
    &-new {
      color: red;
    }
  }

  //css
  .myclass3-new {
    color: red;
  }

17 Each Loop 


  //scss
  @each $member in thom, jonny, colin, phil {
    .bankmanager.#{$member} {
      background: url("images/#{$member}");
    }
  }

  //css
  .bankmanager.thom {
    background: url("images/thom");
  }

  .bankmanager.jonny {
    background: url("images/jonny");
  }

  .bankmanager.colin {
    background: url("images/colin");
  }

  .bankmanager.phil {
    background: url("images/phil");
  }

Each loop within mixin. This is better (but must have 4 arguments).


  //scss
  @mixin band ($a, $b, $c, $d) {
    @each $member in $a, $b, $c, $d{
      .bandmember.#{$member} {
        background: url("images/#{$member}");
        }
      }
    }

  @include band(thom, johnny, colin, phil);

Each loop within mixin with any number of arguments. This is even better. 


  @mixin band2 ($members...) {
    @each $member in $members {
      .bandmember.#{$member} {
        background: url("images/#{$member}");
      }
    }
  }

  @include band2(thom, johnny, colin, phil);

Even better still.


  @mixin band3 ($name, $members...) {
    @each $member in $members {
      .#{$name}.bandmember.#{$member} {
      background: url("images/#{$member}");
      }
    }
  }
  @include band3(webheads, josef, emer, thomas);

This will work, but it is NOT recommended.


  //scss
  @mixin band4 ($name, $members) {
    @each $member in $members {
      .#{$name}.bandmember.#{$member} {
        background: url("images/#{$member}");
      }
    }
  }

  @include band4(webheads, josef emer thomas);

18 Mixins with Optional Arguments

This works, but the order of the arguments given to the mixin is important:


  //scss
  @mixin box($size, $color) {
    width: $size;
    height: $size;
    background: $color;
  }

  .myclass {
    @include box(10px, blue); // Order of args matter
  }

The solution is to use optional arguments (mixin as above). 


  //scss
  .myclass{
    @include box($color:blue, $size:10px);
  }

In addition, default values can be given to mixin arguments. 


  //scss
  @mixin box2($size:100px, $color:red, $display:block) {
    width: $size;
    height: $size;
    background: $color;
    display: $display;
  }

  .box1 {
    @include box2();
  }
  .box2 {
    @include box2($display:inline);
  }
  .box3 {
    @include box2($display: block, $size:200px, $color:green);
  }
  .box4 {
    @include box2(100px, red);
  }

Null may be given as as mixin default argument


  //scss
  @mixin color ($height: null, $width: null)
    {
    height: $height;
    width : $width;
    color: yellow;
  }

  .block {
    @include color($height:10px)
  }

  //css
  .block {
    height: 10px;
    color: yellow;
  }

20 List Maps

List-maps gets rid of global variables. The way to go? Use map-get() to retrieve values from a list-map


    //scss
    $colors: (
    primary: #FA6ACC,
    secondary: #F02A52,
    nav-bar: #09A6E4);

    div {
      background-color: map-get($colors, primary);
      color: rgba(map-get($colors, secondary), 0.4);
    }

    // css
    div {
      background-color: #FA6ACC;
      color: rgba(240, 42, 82, 0.4);
    }

Useful function: map-deep-get() (see here ).


  //scss
  $colors: (
    primary: #FA6ACC,
    secondary: #F02A52,
    nav-bar: (
    primary: green,
    secondary: red)
    );

  @function map-deep-get($map, $keys...) {
    @each $key in $keys {
      $map: map-get($map, $key);
    }
      @return $map;
  }

  .nav {    
      color: map-deep-get($colors, nav-bar, primary);
  }

  //css
  .nav {
    color: green;
  }

21 Pass Code-Block to Mixin

The @content directive allows a code block to be passed to a mixin (or function?). A super-goodie.

Good example (see here)


  /scss
  @mixin media($width) {
    @media only screen and (max-width: $width) {
      @content;
    }
  }

  @include media(320px) {
    background: red;
  }

  //css 
  @media only screen and (max-width: 320px) {
    background: red;
  }

22 Scoping with !default and !global

Note local scoping within the css rule:


  //scss
  $text-color: blue;
   .block {
      $text-color: red;
      color: $text-color;
    }

  .block2{
    color: $text-color;
  }

  //css
  .block {
    color: red;
  }

  .block2 {
    color: blue;
  }

Putting !default at the end of a variable declaration ensures that if a variable is already assiged it will not be over-written, unless the original assignment is to Null (in which case it will be over-written).


  //scss
  $text-color: blue;
  $text-color: red !default;

  .block {
    color: $text-color;
  }

  //css
  .block {
    color: blue;
  }

Create a global variable from within a css rule with !global 


  //scss
  .block {
    $text-color: green !global;
  }

  .block2 {
    color: $text-color;
  }

  //css
  .block2 {
    color: green;
  }

25 Persistently Edit Sass in Chrome

Need to be running Chrome on development machine. Have sass -- watch running (for updates to take effect). Start a server, http-server for example.

From within Chrome, create a workspace as follows:

Development Tools ( Shift-Control-I ) → Sources → Right-Click on App folder → Add app folder to workspace. → Browse to the App folder

Still in Chrome/Sources:

Find <your-app>.scss → Right-Click → Map to network resource

Edit your-app.scss from within Chrome, and save changes to see effects. A goodie.

Underscore and dash in variable names

For historical reasons, variable name (and all other Sass identifiers) can use hyphens and underscores interchangeably. For example, if you define a variable called $main-width, you can access it as $main_width (and vice versa)
